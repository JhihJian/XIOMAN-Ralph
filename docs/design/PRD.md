# React-Agent-Loop PRD

> 持续运行的自主决策 Agent 框架

## 问题定义

### 现有方案的局限

**Ralph 模式**假设：
- 任务可预先枚举为 `prd.json`
- 有明确的「完成」终点
- 执行路径已知

**真实世界的问题**不满足这些假设：
- 目标是「优化代码库」→ 不知道具体要做什么
- 目标是「监控系统健康」→ 没有终点
- 目标是「持续改进产品」→ 路径不可预测

### 本框架的目标

输入是 **Goal（目标）**，不是 Task List（任务列表）。
输出是 **持续工作**，不是「完成」。

核心需求：
1. **不可预知** — 每次迭代前不知道具体要做什么
2. **自主决策** — Agent 自己判断「现在最该做什么」
3. **持续学习** — 跨迭代积累经验，避免重复踩坑

---

## 用户画像

**Agent 开发者**：需要构建长期运行的自主 Agent，而非一次性任务执行器。

**典型场景**：
- 代码库维护 Agent（持续重构、依赖更新、技术债清理）
- 系统运维 Agent（监控、告警响应、故障恢复）
- 产品迭代 Agent（收集反馈、优化体验、A/B 测试）

## 功能需求

### F1: 双循环机制

| 需求 | 描述 |
|------|------|
| F1.1 ReAct 内循环 | Thought → Action → Observation，直到行动完成 |
| F1.2 元认知反思 | 每次迭代结束进行 Pattern 提取和策略蒸馏 |
| F1.3 Fresh Context | 每次迭代从记忆文件重建上下文，不依赖历史对话 |

### F2: 四层记忆系统

| 文件 | 职责 | 更新频率 |
|------|------|----------|
| `goal.md` | 意图层：做什么、不做什么、决策原则、成功标准 | 极少 |
| `semantic.md` | 语义层：验证有效的模式和不变的知识 | 偶尔 |
| `procedural.md` | 程序层：已知会失败的坑和正确做法 | 偶尔 |
| `episodic.md` | 情景层：按时间的事件和决策日志 | 每次迭代 |

**需求**：
- F2.1 四个记忆文件独立存储，支持版本控制
- F2.2 每次迭代自动读取四层记忆
- F2.3 反思阶段自动更新对应记忆层

### F3: 自主决策

| 需求 | 描述 |
|------|------|
| F3.1 优先级公式 | `Priority = Impact × Urgency × Confidence`，可调参可解释 |
| F3.2 候选生成 | 从当前状态和目标推导可能的行动 |
| F3.3 决策理由输出 | 每次选择必须输出可追溯的决策依据 |

### F4: CLI-first 交互

| 需求 | 描述 |
|------|------|
| F4.1 唯一能力是执行 shell 命令 | 不依赖 MCP、API 或特定协议 |
| F4.2 命令输出解析 | 依赖 LLM 理解 stdout/stderr |
| F4.3 跨平台适配 | 处理 Linux/macOS/Windows 命令差异 |

### F5: 状态机驱动

```
SENSE → DECIDE → ACT → REFLECT → (回到 SENSE)
```

**需求**：
- F5.1 SENSE 阶段读取四层记忆 + 执行 CLI 感知外部状态
- F5.2 DECIDE 阶段计算优先级并输出决策理由
- F5.3 ACT 阶段执行 ReAct 循环直到完成或阻塞
- F5.4 REFLECT 阶段更新四层记忆
- F5.5 无终止条件，除非外部中断

## 非功能需求

### N1: 可解释性

- 每个决策可追溯到优先级计算
- 每次反思可追溯到具体事件
- 记忆文件人类可读

### N2: 可调参

- 优先级公式权重可配置
- 反思深度可配置
- 记忆保留策略可配置

### N3: 容错

- 命令执行失败不中断循环
- 反思失败降级为简单日志
- 记忆写入失败有备份机制

---

## 成功标准

**必须满足**：
1. 给定一个 Goal，Agent 能自主决策下一步行动
2. 跨迭代能积累经验（procedural.md 中有 Anti-Pattern 记录）
3. 每次迭代有明确的决策理由输出

**期望达到**：
1. Priority 公式在 A/B 测试中优于纯 LLM 判断
2. semantic.md 中的 Pattern 能在后续迭代中被检索和复用
3. 运行 100 次迭代后，错误率明显下降

**超出预期**：
1. 支持动态生成 SubAgent
2. 多 Agent 协作共享记忆
3. 量化「越用越聪明」指标

---

## 边界

**做**：
- 单 Agent 持续运行
- CLI-first 工具调用
- 四层记忆管理
- 元认知反思

**不做（Future Work）**：
- SubAgent 架构
- 多 Agent 协作
- 持续学习验证机制
- GUI/IDE 集成
